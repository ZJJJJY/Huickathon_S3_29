# 三分钟热度 · 技术设计文档 (design.md)

> Hackathon 题目：AI 体验 —— 刷到懂你的瞬间
> 项目代号：三分钟热度（Three-Minute Spark）
> 时间约束：12 小时
> 端形态：**Mobile-First Web（仅手机端）**

---

## 1. 项目概述

### 1.1 问题陈述

**痛点：**
用户在短视频平台（抖音、B站、小红书）刷到一个看起来很酷的爱好（攀岩、陶艺、滑板、飞盘…）时，会产生即时的"我也想试试"的冲动。但从冲动到行动之间存在一条断裂的链路：

1. **搜索成本高**：不知道怎么描述这个爱好、搜什么关键词
2. **信息碎片化**：搜出来的内容散落在多个平台，要自己拼装"装备篇 + 入门篇 + 避坑篇"
3. **缺乏个性化判断**：教程是写给所有人的，没人告诉你"以你的预算 / 时间 / 城市，这个爱好能不能玩、怎么玩"
4. **冲动窗口短**：一旦放下手机或被其他内容分散，"三分钟热度"就消失了

最终结果：90% 的爱好冲动死在搜索阶段，用户回到刷视频，错过一次拓展生活的机会。

**目标用户：**
- 18–35 岁，活跃于短视频平台
- 对新事物有好奇心，但不擅长 / 不愿意做大量预研
- 决策风格偏"先看看值不值得开始，再决定要不要投入"
- 不是已经入坑的发烧友（他们不需要这个产品）

### 1.2 解决方案

**一句话：**
一个在用户产生爱好冲动的瞬间，通过 4 个问题快速理解你，并由 **Web Search Agent** 实时汇总真实互联网内容生成一份结构化、个性化"入坑决策报告"的轻量 AI 工具。

### 1.3 核心价值主张

| 维度 | 自己搜索 | 三分钟热度 |
|---|---|---|
| 信息形态 | 一堆链接，需自行整合 | 一份结构化报告 |
| 信息来源 | 自己点开 N 个 App | Agent 汇总多平台真实内容 |
| 信息可信度 | 自己判断 | 基于真实搜索结果，规避 LLM 幻觉 |
| 个性化 | 通用内容 | 基于预算 / 时间 / 地域 / 入门期望生成 |
| 决策导向 | 偏教程（"怎么做"） | 偏决策（"值不值得开始 + 怎么开始"） |
| 耗时 | 15–30 分钟 | 30 秒问卷 + 即时生成 |
| 去魅化 | 滤镜内容居多 | 主动呈现常见放弃节点、真实预期 |

**关键差异化：**
- 不是 LLM 凭训练数据"凭空生成"，而是 **Agent 实时检索 + 汇总真实内容**，降低幻觉
- 不是"爱好百科"，而是"爱好决策器"——回答"我能不能、值不值得开始"

### 2.2 技术栈选择

| 层 | 选型 | 理由 |
|---|---|---|
| **前端框架** | Next.js 14 (App Router) | 一体化前后端，省去跨服务配置；Vercel 一键部署 |
| **UI 组件** | Tailwind CSS + shadcn/ui | 12h 内最快搭出专业感卡片式 UI |
| **城市搜索框** | shadcn `Command` + `Popover` | 内置搜索过滤，支持拼音/汉字模糊匹配 |
| **状态管理** | React useState + URL Query | 单页流程无需 Redux；问卷答案存 URL 便于分享 |
| **后端** | Next.js API Routes (Node 20) | 与前端同构，免部署额外服务 |
| **Agent 框架** | **Claude 3.5 Sonnet + Tool Use** | 原生工具调用稳定、支持流式、JSON 输出可控 |
| **Web 搜索** | **Serper API（主）/ Tavily（备）** | 国内访问稳定，覆盖知乎/小红书/B站等中文来源 |
| **数据存储** | JSON 文件 + LocalStorage | 12h 内不引入数据库；历史/缓存存浏览器即可 |
| **部署** | Vercel | 5 分钟上线，自带 HTTPS，扫码体验流畅 |

**显式不引入的东西：**
- 用户登录系统：演示不需要
- 数据库：MVP 阶段没有跨设备同步需求
- 直接爬抖音/小红书：反爬不稳定，改走通用搜索引擎间接索引
- 图片内容理解：Agent 仅处理文字，节省时间和成本
- PC 端适配：仅做手机端

### 2.4 API 设计

#### POST `/api/generate`

主接口：生成结构化入坑报告。

**Request:**
```json
{
  "hobby": "攀岩",
  "profile": {
    "region_code": "SH",
    "region_name": "上海",
    "budget": "medium",
    "time_per_week": "weekly",
    "expected_onboarding": "1month"
  }
}
```

**字段枚举：**
- `region_code`: 来自城市枚举表（见 4.2）
- `budget`: `low` | `medium` | `high`
- `time_per_week`: `fragment` | `weekly` | `deep`
- `expected_onboarding`: `1week` | `1month` | `3month` | `flexible`

**Response (Server-Sent Events 流式):**
```
event: meta
data: { "hobby": "攀岩", "category": "体能体验型", "theme_color": "#FF6B35" }

event: section
data: { "key": "expectation", "content": { ... } }

event: section
data: { "key": "barrier", "content": { ... } }

event: section
data: { "key": "cost", "content": { ... } }

event: section
data: { "key": "action_plan", "content": { ... } }

event: sources
data: { "urls": ["https://www.zhihu.com/...", "https://www.xiaohongshu.com/..."] }

event: done
data: {}
```

每个 section 的完整结构同上一版（expectation / barrier / cost / action_plan），略。

#### GET `/api/random`

返回一个预置 demo 爱好（用户不知道填什么时使用）。

**Response:**
```json
{
  "hobby": "陶艺",
  "preview_text": "想试试用手捏一个杯子？",
  "is_demo_cached": true
}
```

#### GET `/api/cities?q=<query>`

城市搜索（前端可选直接读静态 JSON，无需走后端）。

**Response:**
```json
[
  { "code": "SH", "name": "上海", "pinyin": "shanghai", "tier": "T1" },
  { "code": "BJ", "name": "北京", "pinyin": "beijing", "tier": "T1" }
]
```

> **MVP 决定：** `/api/session/save` 在 MVP 阶段**不实现**。报告历史完全用前端 LocalStorage 管理，省一个接口的开发和测试时间。

### 3.2 用户故事

- As a **刷视频时被某个爱好击中的用户**, I want to **30 秒内得到一份基于我个人情况且来自真实互联网内容的入坑判断**, so that **我不用打开 5 个 App 自己拼信息，也不用担心 AI 编造信息**。

- As a **想拓展生活但不知道该尝试什么的用户**, I want to **点一下"随机"就能看到一个完整的爱好介绍**, so that **我能在探索中发现自己可能喜欢的方向**。

- As a **已经搜过但觉得信息太散的用户**, I want to **一个页面看到门槛、成本、第一步行动，并附带来源链接**, so that **我能立即判断要不要花周末去尝试，也能在感兴趣时点进原文深挖**。

- As a **担心自己三分钟热度的用户**, I want to **看到"大多数人在第几次放弃"的真实信息**, so that **我对自己的预期更合理，不会因为浪漫想象买一堆装备然后吃灰**。

## 4. 数据模型

### 4.1 实体关系图（ERD）

└──────────────────┘        └─────────┬──────────┘
                                      │ *
                                      │
                                      ▼ 1
                            ┌────────────────────┐
                            │  HobbyTemplate     │
                            │  (Seed JSON)       │
                            └────────────────────┘

┌──────────────────┐
│  City (Enum)     │  ← 静态 JSON，前端直接加载
│  (Seed JSON)     │

### 4.2 关键数据结构

#### City（枚举：城市）

```typescript
interface City {
  code: string;       // "SH" / "BJ" / "CD"
  name: string;       // "上海" / "北京" / "成都"
  pinyin: string;     // "shanghai" / "beijing" / "chengdu"
  tier: 'T1' | 'T1.5' | 'T2' | 'T3';  // 城市分级，影响 LLM 对当地资源的判断
  province?: string;
}
```

**城市数据获取策略：**
- **首选方案**：使用现成的开源 JSON（GitHub 上有现成的"中国行政区划"数据，如 `china-area-data` 或 `pca-code.json`），筛选地级市以上共约 300 个。
- **备选方案**：高德地图开放平台的"行政区域查询" API 一次性导出（注册即可，免费）。
- **手撸方案**：直接写一个 50-100 个主流城市的精简表（一线 + 新一线 + 二线），覆盖 90% 用户。
- **hackathon 推荐**：取开源 JSON，简化字段后做成 `/data/cities.json`，前端直接 import。

**前端交互：**
- 使用 shadcn `Command` 组件实现可搜索下拉框
- 支持汉字输入（"上"匹配上海、上饶、上虞）
- 支持拼音输入（"sh" 匹配所有 sh 开头的城市）
- 默认展示热门一线城市（北上广深 + 成都/杭州/武汉 等）

```typescript
interface Report {
  id: string;

#### HobbyTemplate（Seed JSON）

```typescript

## 6. 风险与取舍

### 6.1 技术风险

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| **Agent 响应时间过长（>30s）** | 高 | 高 | 流式输出 + 加载页友好提示 + 限制搜索结果数量（top 5-8） |
| **搜索结果质量参差，agent 引用低质来源** | 高 | 中 | prompt 中明确要求优先 zhihu/xhs/bilibili 域名 + 在结果中展示来源域名让用户自判 |
| **Serper / Tavily API 限流或费率问题** | 中 | 高 | 准备双家 API（主备切换）+ 缓存最近 10 次结果 |
| **Agent 调用失败 / JSON 解析失败** | 中 | 高 | 自动 fallback 到同类别 demo 缓存报告 |
| **小众爱好搜不到优质来源** | 中 | 中 | prompt 中强制 agent 说明"信息有限"而非编造 |
| **演示现场网络不稳定** | 低 | 极高 | 全套 3 个 demo 报告本地缓存，纯前端可演示 |
| **20字限制导致用户表达不全** | 低 | 低 | 前端 placeholder 提示用户精炼描述 |

### 6.2 范围取舍

**明确砍掉 / 推迟的功能：**

| 功能 | 决定 | 原因 |
|---|---|---|
| 直接爬抖音 / 小红书 | 砍掉 | 反爬不稳定，改走 Serper 间接索引 |
| Agent 图片内容理解 | 砍掉 | 成本高、延迟大，对入坑指南文字足够 |
| 视频链接 / 截图识别输入 | 砍掉 | 技术复杂，文字输入足够演示 |
| 用户登录 / 跨设备同步 | 砍掉 | 演示不需要 |
| 数据库 / `/api/session` | 砍掉 | LocalStorage 足够，省一整套接口 |
| PC 端 / 桌面端适配 | 砍掉 | 仅 mobile-first |
| 报告底部对话式追问 | 推迟到加分项 | 实现复杂，先保证主流程 |
| 地图 API 接入（具体场馆） | 砍掉 | 让 agent 在 prompt 中描述性提及即可 |
| 复杂个性化（>4 维问卷） | 砍掉 | 4 题已能产生明显差异化 |

### 6.3 降级方案

**核心功能失败时的备选层级：**

1. **Agent 调用超时（>30s）** → 中止搜索，让 Claude 直接基于训练知识生成（标记"快速模式"），并提示用户"信息可能不够新鲜"
2. **Agent 输出非合法 JSON** → 后端尝试修复 / 提取 ```json 块，失败则返回缓存的同类别 demo
3. **Serper API 全部失败** → 切换 Tavily 备用；若都失败，纯 Claude 模式（接受幻觉风险，但保证有结果）
4. **用户输入极冷门词** → Agent 在 prompt 中被要求："如果搜索结果少于 3 条相关内容，明确告知用户并推荐尝试随机抽取"
5. **整个 `/api/generate` 失败** → 前端的"🎲 随机"按钮始终可用（命中预生成 demo 缓存，纯前端可达）
6. **演示现场断网** → 切换到本地 mock 模式，3 个预置 demo 完全本地化跑通

**核心原则：演示路径必须保证 100% 可达**——即使所有外部依赖失效，扫码 → 问卷 → 随机 → 看到精心打磨的 demo 报告，整条体验链不能断。

**每个 demo 准备 3 套画像版本**（覆盖最常见的问卷组合），确保现场任何人扫码都能得到"像在说我"的报告：
- 低预算 + 碎片时间 + 一线城市

### 7.2 观众在游园会怎么体验

**现场动线（每位观众约 90 秒）：**

```
[展位摆放二维码 + 一句话标语]
      ▼
观众扫码 → 落地页（看到"刷到一个爱好，30 秒看它适不适合你"）
      ▼
[志愿者引导]："想想你最近刷到过想试试的爱好，没有也没关系"
      ▼

      ▼

   ┌─ 路径 A：观众有想了解的 → 输入爱好（≤20字）→ Agent 实时检索
   │        （展示"AI 在帮我搜真实内容"的过程，避免黑箱感）
   │
   └─ 路径 B：观众想不到 → 点击🎲随机 → 命中预置 demo
            （保证演示流畅 + 内容质量稳定）
      ▼
报告页展示（重点观察观众反应：放大看哪个板块？是否点进来源链接？）
      ▼

      ▼
观众可保存 / 再来一个
```

**展位物料：**
- 大屏轮播：3 个 demo 报告的"惊艳板块"截图（放弃节点、成本档位、行动计划）
- 桌面立牌：二维码 + slogan + 一行说明
- 备用平板：现场扫码不便时，志愿者手持设备引导体验

**评委 demo 路径（3 分钟版本）：**
1. 0:00 — 一句话痛点："你有多少次刷视频想学新东西，最后什么都没做？"
2. 0:30 — 扫码 → 现场填 4 题问卷（城市搜索时让评委自己输入）
3. 1:00 — 输入评委自己想到的爱好，展示 Agent 检索过程（"它正在去知乎、小红书帮我找答案"）
4. 1:30 — 报告生成后，解说 4 个板块，重点强调"基于评委画像 + 真实来源"
5. 2:30 — 点击🎲展示预生成 demo，强调演示稳定性
6. 3:00 — 收尾："我们让一个 30 秒的冲动，变成一个可以执行的决定。"

**风险预案：**
- 若现场网络不稳：志愿者引导观众走"随机抽取"路径（纯前端缓存，零依赖）
- 若 Agent 调用过慢：加载页有趣味文案（"正在翻 200 篇笔记…"），缓解等待焦虑
- 若 Agent 服务异常：后端自动 fallback 到缓存报告
- 若观众输入冷门爱好生成质量差：志愿者主动建议"试试随便抽一个，效果更稳定"

## 8. 加分项与扩展规划

> 以下功能**不在 MVP 范围内**，仅在 12h 主流程跑通且有富余时间时考虑。每项都附带实现注意事项。

### 8.1 流式输出与"Agent 工作可视化"

**功能：** 报告生成过程中，前端实时显示 agent 的工作步骤（"正在搜索…找到 12 条…正在汇总真实预期…"），让用户感受到"AI 真的在帮我干活"。

**实现要点：**
- 后端通过 SSE 推送 agent 各阶段状态
- 前端用类似终端打字机的效果渲染
- 注意：每个步骤至少展示 0.5s，否则太快用户感受不到价值

**注意事项：**
- 不要露出 prompt 细节，只展示用户能理解的语言（"搜索"而非"调用 search_web 工具"）
- 失败步骤要优雅处理（"搜索遇到问题，正在尝试备用方案…"）

### 8.2 历史记录

**功能：** 用户查询过的爱好自动存入 LocalStorage，落地页/报告页底部展示"你最近看过的"，方便回访。

**实现要点：**
- LocalStorage key: `tms_history`，存最近 5 条
- 显示形式：横向滑动卡片，缩略图 + 爱好名
- 点击直接展示缓存报告，无需重新请求

**注意事项：**
- 注意隐私感知（不要让用户觉得被监视）
- 提供"清空历史"入口

### 8.3 报告板块预设追问

**功能：** 每个板块底部 2-3 个预设追问按钮（如"附近哪里能体验"、"还有什么替代爱好"），点击触发新一轮 agent 调用。

**实现要点：**
- 预设问题由 agent 在生成主报告时一并产出（避免单独再调一次）
- 追问结果作为补充内容追加在板块下方，不替换原内容

**注意事项：**
- 不做完全开放的对话框，避免用户问出 agent 答不好的问题
- 每个板块预设问题需与该板块强相关，避免泛化

### 8.4 分享卡片生成

**功能：** 生成一张可分享到微信/朋友圈的报告摘要图片，包含爱好名、4 板块关键信息、二维码。

**实现要点：**
- 用 `html-to-image` 或 `satori` 库前端直接生成 PNG
- 图片尺寸 1080×1920（朋友圈友好比例）
- 模板设计要简洁有传播感

**注意事项：**
- 字体加载要做 fallback，避免生成失败
- 二维码指回产品主页，形成传播闭环

### 8.5 视频链接 / 截图识别

**功能：** 用户粘贴抖音/B站视频链接，或上传截图，自动识别爱好类型并填入。

**实现要点：**
- 链接解析：用 oembed API 或简单 HTML scrape 提取视频标题/描述
- 截图识别：调用 Claude vision 或 GPT-4V
- 识别结果作为预填入，用户可修改

**注意事项：**
- 这是 P3 级别加分项，对应大量边界 case（识别失败、版权问题、加载慢）
- 仅在主流程完全稳定后才能动手，否则风险传染

### 8.6 多语言支持

**功能：** 报告支持英文输出，适配国际场景。

**注意事项：**
- 12h hackathon 完全没必要做
- 如未来要做，agent 的 search query 需根据语言切换搜索域

### 8.7 用户反馈收集

**功能：** 报告底部"这份报告对你有用吗"评分入口，收集质量反馈。

**实现要点：**
- 简单的👍/👎按钮
- 评分数据存到 Vercel KV 或简单的 Google Form

**注意事项：**
- hackathon 阶段不需要，但赛后想做产品化时这是第一步要补的

### 8.8 通用注意事项

- **任何加分项都不应抢占主流程时间。** MVP 没跑通之前，加分项一行代码都不要写。
- **加分项的失败不能拖累主流程。** 例如分享功能失败不能阻塞报告展示。
- **加分项的优先级判断标准：** 演示时评委一眼能感知到 > 用户实际会用到 > 代码量少但效果明显。按这个顺序，**8.1 流式可视化** 是最值得做的加分项。
